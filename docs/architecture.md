# 基金管理项目架构设计

## 1. 项目目标

建设一个面向个人投资者的基金管理系统，支持以下核心能力：

- 录入和维护个人基金持仓
- 根据基金净值、估值及持仓成本实时推测当日收益和累计收益
- 管理自选基金池
- 通过数据看板展示持仓分布、收益走势、风险暴露和基金表现
- 将数据库、缓存、反向代理和最终运行环境统一部署在远程服务器 `<server-ip>`
- 本地支持完整开发运行，生产环境统一部署在远程服务器

## 3. 系统架构

系统采用前后端分离架构。

### 3.1 模块划分

1. `frontend-web`
   - 用户登录
   - 持仓管理
   - 自选管理
   - 数据看板
   - 收益明细查询

2. `backend-api`
   - 用户认证与权限管理
   - 基金基础信息管理
   - 持仓交易记录管理
   - 收益估算服务
   - 自选服务
   - 看板聚合服务
   - 定时同步基金净值/估值

3. `mariadb`
   - 用户数据
   - 基金信息
   - 持仓交易流水
   - 净值快照
   - 自选关系

4. `redis`
   - 基金实时估值缓存
   - 热门看板数据缓存
   - 登录辅助缓存

### 3.2 核心数据流

1. 用户在前端录入买入基金、份额、买入金额、手续费和时间。
2. 后端保存交易流水，按基金维度聚合当前持仓成本与份额。
3. 定时任务抓取基金净值和盘中估值，写入缓存与历史快照。
4. 收益估算服务基于“当前份额 × 当前估值/净值”计算持仓市值，并与成本比较得到收益。
5. 看板服务聚合资产、收益率、基金分类分布、日收益排行等指标供前端展示。

## 4. 业务设计

### 4.1 持仓管理

支持：

- 新增买入记录
- 新增卖出记录
- 查看当前持仓
- 查看交易明细
- 计算持仓成本、持仓份额、持仓收益、累计收益率

### 4.2 收益估算逻辑

收益分两类：

1. `实时估算收益`
   - 适用场景：交易日盘中
   - 数据基础：第三方基金估值、上一交易日确认净值、持仓份额
   - 计算方式：
     - 估算市值 = 当前持仓份额 × 当前估算净值
     - 估算收益 = 估算市值 - 当前持仓成本
     - 当日估算收益 = 当前持仓份额 × (当前估算净值 - 昨日净值)

2. `结算收益`
   - 适用场景：收盘后或净值更新后
   - 数据基础：基金正式净值
   - 计算方式：
     - 结算市值 = 当前持仓份额 × 最新正式净值
     - 结算收益 = 结算市值 - 当前持仓成本

说明：基金申购、赎回的确认存在 T+1 或更长时滞，第一阶段按“用户录入即参与测算”处理，后续再补确认日模型。

### 4.3 自选功能

支持：

- 添加基金到自选列表
- 取消自选
- 查看自选基金估值、涨跌幅、近 1 周/1 月表现
- 从自选快速加入持仓

### 4.4 数据看板

第一阶段看板指标：

- 总投入金额
- 当前持仓市值
- 累计收益额
- 累计收益率
- 今日预估收益
- 持仓基金数量
- 基金类型分布
- 收益排名 Top N
- 亏损排名 Top N
- 近 7 天收益趋势

## 6. 接口设计原则

- 前后端统一使用 RESTful API
- 返回结构统一：`code`、`message`、`data`
- 所有时间统一使用 `Asia/Shanghai`
- 金额、净值使用 `BigDecimal`
- 涉及收益率统一保留 4 位小数，金额保留 2 位小数

## 8. 当前服务器适配结论

基于实际服务器巡检，确认以下事实：

- 服务器系统为 `Debian 12`
- 已存在 `OpenJDK 17`、`Node.js 22`、`Nginx`
- 当前 Nginx 已启用站点已被停用，后续由本项目独占 `80` 端口入口
- 当前服务器已经安装并运行 `MariaDB`、`Redis`、`Maven`

因此第一阶段采用以下适配策略：

- 后端运行时统一使用现有 `Java 17`
- 数据库采用 Debian 原生支持更好的 `MariaDB 10.11`
- 后端服务监听 `127.0.0.1:18080`，仅允许 Nginx 反向代理访问
- 前端静态资源由 Nginx 直接托管在根路径 `/`
- 后端 API 统一通过 `/api/` 暴露
